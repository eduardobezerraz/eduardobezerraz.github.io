O que é autenticação?

Autenticação é o processo de verificar quem é alguém. É fácil confundir com autorização, mas são coisas diferentes:

  • Autenticação confirma que você é quem diz ser.
  • Autorização decide o que você tem permissão de fazer.

Existem três tipos de fator de autenticação:

  • Algo que você sabe :senha, resposta de pergunta de segurança.
  • Algo que você tem :celular, token físico, app gerador de código.
  • Algo que você é :biometria.

É comum que aplicações utilizem apenas o primeiro fator (senha). Falhas de autenticação são especialmente graves porque, se um atacante conseguir burlar esse processo, ele ganha acesso a tudo que aquela conta pode fazer e se a conta comprometida for de um administrador, o estrago pode ser maior.

Falhas no login por senha

No login por senha, o sistema assume que só quem conhece a senha pode ser o dono da conta. Mas isso pode ser contornado de duas formas: brute-force (adivinhar credenciais por tentativa e erro) e falhas de lógica que permitem pular a autenticação inteira.

Brute-force de usuário e senha

Brute-force não precisa ser aleatório. Conhecendo padrões comuns, um atacante reduz drasticamente o espaço de busca:

Padrões previsíveis de usuário e senha 
# usernames corporativos podem seguir um padrão óbvio:
joao.silva@company-vulnerable-site.com
maria.santos@company-vulnerable-site.com

# usuários tentam encaixar a própria senha numa política
# em vez de criar uma senha de verdade aleatória:
company123      →  Company123!     # satisfaz "1 maiúscula + 1 símbolo"
company123      →  Company1234!    # renovação periódica de senha

Isso significa que com uma wordlist bem pensada, baseada em comportamento humano, é muito pode ser muito mais eficiente que tentar todas as combinações possíveis de caracteres.

Enumeração de usuários

Enumeração de usuários é quando você consegue determinar se um usuário existe ou não, quando o site, sem querer, revela se um determinado nome de usuário existe na base. Isso reduz drasticamente o trabalho de um atacante, porque ele só precisa focar em adivinhar a senha de contas que ele já sabe que existem.

Os três sinais mais comuns de enumeração:

1. Código de status diferente

Python — comportamento vulnerável no backend 
def login(username, password):
    user = db.find_user(username)

    if user is None:
        return Response(status=404)  #  usuário não existe

    if user.password != hash(password):
        return Response(status=401)  #  usuário existe, senha errada

    return Response(status=200)

Dois códigos diferentes (404 vs 401) já dizem ao atacante exatamente quais usernames existem, sem nem precisar acertar a senha.

2. Mensagem de erro diferente

Mensagens que revelam a existência do usuário 
# usuário não existe:
"Usuário não encontrado."

# usuário existe, senha errada:
"Senha incorreta."

# o correto é uma mensagem genérica idêntica nos dois casos:
"Usuário ou senha incorretos."
🔍 Até um espaço em branco ou ponto final fora de lugar transforma duas mensagens "idênticas" em mensagens distinguíveis no HTML — mesmo que a diferença não apareça visualmente na tela.

3. Tempo de resposta diferente

Se o sistema só verifica a senha depois de confirmar que o usuário existe, essa etapa extra introduz um delay perceptível:

Python — verificação que vaza tempo de resposta 
def login(username, password):
    user = db.find_user(username)
    if user is None:
        return fail()  # 10 ms se o user não existe

    # bcrypt.checkpw checa o hash da senha e é lento
    if bcrypt.checkpw(password, user.password_hash):
        return success()
    return fail()  # 120 ms se o user existir

Um atacante pode tornar essa diferença ainda mais óbvia mandando uma senha gigante de propósito, já que o tempo extra de processamento fica mais perceptível.

Proteções fracas contra brute-force

As duas defesas mais comuns são bloqueio de conta (depois de N tentativas falhas) e rate limiting por IP. Nenhuma delas, isoladamente, é suficiente para impedir brute-force.

Bloqueio de conta — burlado com poucas senhas e muitos usuários

Se o limite é, por exemplo, 3 tentativas antes de bloquear a conta, um atacante pode inverter a lógica do ataque: em vez de tentar muitas senhas contra um usuário, ele tenta poucas senhas contra muitos usuários, sem ultrapassar o limite configurado para cada conta:

Estratégia que nunca aciona o bloqueio 
usuarios = ["joao", "maria", "carlos", ... ]  # centenas de contas
senhas   = ["123456", "senha123", "g1its2024"] # só 3 (dentro do limite)

for senha in senhas:
    for usuario in usuarios:
        tentar_login(usuario, senha)
        # cada conta recebe no máximo 3 tentativas e nunca bloqueia
        # só precisa UM usuário usar UMA dessas senhas pra funcionar

Rate limiting por IP — burlado trocando o IP

Se a defesa é só "bloquear o IP depois de X tentativas", o atacante pode distribuir as requisições entre vários IPs (usando proxies, VPN, ou até manipulando headers como X-Forwarded-For, se o servidor confiar cegamente neles).

🔍 Uma falha comum: zerar o contador de tentativas falhas quando o dono do IP faz login com sucesso. Isso permite que o atacante "resete" o contador de propósito, intercalando o próprio login válido a cada poucas tentativas de brute-force contra outras contas.

Falhas em autenticação multifator

Autenticação de dois fatores (2FA) combina algo que você sabe (senha) com algo que você tem (código de um app ou SMS). É mais segura que a senha sozinha se for implementada corretamente.

💡 2FA por e-mail pode ser uma verificação não tão efetiva, pois acessar o código de verificação no e-mail pode só exigir a mesma senha que o atacante já teve acesso (caso o atacante já tenha comprometido as mesmas credenciais utilizadas para acessar o e-mail), se a senha do usuário for a mesma. É como verificar o mesmo fator duas vezes.

1. Pular a segunda etapa por completo

As vezes após a senha verificada na etapa 1 e durante a verificação do código na etapa 2, o usuário já está "meio logado", e a aplicação não verifica se a etapa 2 foi realmente concluída com sucesso. Se a implementações for mal feita, pode ser possível acessar diretamente uma página que devia aparecer após login completo (pulando etapas):

Pulando direto para a página protegida 
# etapa 1: login normal com senha feito com sucesso
POST /login-step-1
username=carlos&password=correto123


#  servidor já cria uma sessão "parcial" pro usuário
#  e pede para que o usuário faça a segunda etapa

#  o atacante pode ignorar a etapa 2 (código) e tentar direto:
GET /minha-conta
#  algumas implementações deixam de verificar se a etapa 2 foi concluída
#  elas apenas verificam se existe uma sessão

2. Cookie de sessão trocável entre a etapa 1 e a etapa 2

Esse é o bypass mais perigoso. Depois da senha, o servidor associa a sessão a um cookie indicando de quem é a conta, e a segunda etapa pode confiar demais nesse cookie:

Fluxo legítimo de login em 2 etapas 
# 1. login com a própria senha:
POST /login-step-1
username=atacante&password=minha-senha

# 2. servidor responde com um cookie identificando a conta:
Set-Cookie: account=atacante

# 3. etapa final usa esse cookie pra saber de quem é o código:
POST /login-step-2
Cookie: account=atacante
verification-code=123456
O bypass: trocar o cookie antes da etapa final 
# o atacante loga normalmente com a própria conta (etapa 1)...
POST /login-step-1
username=atacante&password=minha-senha
# ...mas troca o valor do cookie manualmente antes da etapa 2:

POST /login-step-2
Cookie: account=vitima-user
verification-code=000000

# se o código também puder ser brute-forçado (4-6 dígitos),
# o atacante entra na conta da vítima sem precisar saber a senha dela

3. Código de verificação fraco contra brute-force

Um código de 6 dígitos tem só 1 milhão de combinações e torna-se viável um brute-force caso não existam limites de tentativas. Deslogar o usuário depois de N tentativas, por si só, pode não ser suficiente, porque um ataque automatizado reinicia o processo de login do zero a cada bloqueio.

Outros mecanismos: "lembrar de mim" e reset de senha

Cookie de "lembrar de mim" previsível

Se o cookie é construído a partir de valores estáticos e previsíveis (nome de usuário + timestamp, por exemplo), um atacante que entende a fórmula pode tentar forjar o cookie de outros usuários:

Python — cookie "remember me" mal construído 
import base64

def generate_remember_token(username):
    raw = f"{username}:{timestamp_do_login}"
    return base64.b64encode(raw.encode())
    # base64 não é criptografia! É só uma codificação reversível
    # qualquer pessoa pode decodificar e descobrir a fórmula
💡 Base64 não fornece confidencialidade, é só uma forma de representar bytes como texto, igual ao URL encoding. Mesmo um hash sem salt pode ser quebrado: hashes de senhas comuns já vazadas estão disponíveis publicamente, em alguns casos, basta consultar bases públicas de hashes conhecidos.

Reset de senha por parâmetro adivinhável

Uma implementação insegura consiste em:

Reset de senha vulnerável 
https://g1its-vulnerable-site.com/reset-password?user=vitima-user

# trocar o valor da url para o endpoint da tela de "definir nova senha",
# pode levar ao acesso sem precisar nunca ter recebido um e-mail real

O jeito correto é gerar um token de alta entropia, sem pistas sobre qual conta ele pertence:

Python — token seguro de reset 
import secrets

def generate_reset_token():
    return secrets.token_hex(32)
    # token longo, aleatório, sem relação visível com o usuário
    # precisa expirar rápido e ser invalidado após o uso

Como prevenir

Não existe uma correção única. A autenticação precisa de várias camadas trabalhando juntas:

  • Nunca envie credenciais em texto puro. HTTPS sempre, incluindo redirecionamento automático de HTTP.
  • Mensagens e tempos de resposta idênticos. Login errado deve parecer igual do ponto de vista do cliente, não importa o motivo do erro.
  • Rate limiting bem implementado. Por IP, conta e outros identificadores, aplicando atrasos progressivos ou CAPTCHA quando apropriado.
  • Tokens de alta entropia para reset de senha e "lembrar de mim", nunca valores previsíveis ou só codificados em Base64.
  • 2FA com fator realmente diferente Utilize um segundo fator independente, como aplicativos autenticadores ou chaves de segurança. Evite depender apenas do e-mail quando ele é protegido pelas mesmas credenciais da conta.
  • Revalide a sessão a cada etapaValide o estado da autenticação em todas as etapas, verificando no servidor se o usuário realmente concluiu o fluxo (senha, 2FA etc.), em vez de confiar apenas na existência de um cookie ou sessão.

Exemplo de login com mensagens e tempo de resposta uniforme:

Python — login com mensagens e tempo uniformes 
import bcrypt

# Hash bcrypt válido usado para manter o tempo uniforme
DUMMY_HASH = b"$2b$12$A0DL0s4vW3j2j7GmPzQJ1e7Wk8Y8J3kHjGkJr9z5M5m3xR7k8uL6."

def login(username, password):
    # procura o usuário
    user = db.find_user(username)

    # # verifica um hash independentemente de o usuário existir
    hash_to_check = user.password_hash if user else DUMMY_HASH

    # executa o bcrypt nos dois casos
    senha_ok = bcrypt.checkpw(
        password.encode(),
        hash_to_check
    )

    # mesma mensagem e mesmo status para ambos os erros
    if user is None or not senha_ok:
        return Response(
            "Usuário ou senha incorretos.",
            status=401
        )

    # login concluído
    return Response(
        "Login bem-sucedido.",
        status=200
    )
✅ Rodar o hash mesmo quando o usuário não existe reduz diferenças de tempo de resposta entre os casos, dificultando ataques de enumeração baseados em temporização. O servidor sempre faz a mesma quantidade de trabalho, então não há diferença mensurável entre "usuário não existe" e "senha errada".
