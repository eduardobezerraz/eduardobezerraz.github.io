O que é path traversal?

Path traversal (também chamado de directory traversal) é uma vulnerabilidade que permite a um atacante ler arquivos arbitrários no servidor, possibilitando o acesso a arquivos que a aplicação não deveria expor.

⚠️ Em casos mais graves, o atacante consegue também escrever arquivos no servidor, modificando o comportamento da aplicação e chegando ao controle total da máquina.

Como funciona na prática

Imagine uma loja online que exibe imagens de produtos. Para carregar cada imagem, ela faz uma requisição assim:

HTML da página 
<img src="/loadImage?filename=218.png">

O servidor recebe o parâmetro filename e busca o arquivo dentro de uma pasta base — por exemplo, /var/www/images/. Então ele lê o caminho final:

/var/www/images/218.png # Esse é o comportamento esperado.

O problema: a aplicação não valida o que chega no parâmetro filename. Um atacante pode mandar isso:

Requisição maliciosa 
https://site-vulneravel.com/loadImage?filename=../../../etc/passwd

O servidor monta o caminho /var/www/images/../../../etc/passwd e lê exatamente isso. Veja o que acontece passo a passo:

/var/www/images/ ← ponto de partida
../ ← sobe para /var/www/
../ ← sobe para /var/
../ ← sobe para / (raiz)
etc/passwd ← arquivo lido!
💡 ../ é uma sequência válida em sistemas de arquivos que significa "suba um nível no diretório". No Windows, funciona tanto ../ quanto ..\.

No Windows também funciona

O mesmo ataque em um servidor Windows ficaria assim:

Windows 
https://site-vulneravel.com/loadImage?filename=..\..\..\windows\win.ini

Obstáculos e como contorná-los

Muitas aplicações tentam se proteger bloqueando ou removendo ../ da entrada do usuário. Mas essas defesas costumam ser frágeis e têm bypasses conhecidos.

1. Caminho absoluto

Se a aplicação apenas bloqueia ../ mas aceita caminhos absolutos, você pode referenciar o arquivo diretamente:

Sem precisar de ../ 
filename=/etc/passwd

2. Sequências aninhadas

Se a aplicação remove ../ mas não de forma recursiva, sequências aninhadas se reconstroem sozinhas após a remoção:

Bypass por sequência aninhada 
filename=....//....//....//etc/passwd

# remover "../" de "....//":
# ....// - ../  =  ../

3. Encoding na URL

O servidor web pode estar sanitizando a URL, mas a aplicação processa o valor já decodificado. Encodar os caracteres contorna a sanitização:

No padrão URL Encoding, caracteres especiais são representados em hexadecimal precedidos por %:

URL encoding 
# ../  →  URL encoded
%2e%2e%2f

# ../  →  double URL encoded
%252e%252e%252f

# variações não-padrão que podem funcionar
..%c0%af
..%ef%bc%8f

4. Pasta base obrigatória no início

Algumas aplicações exigem que o filename comece com a pasta base. É possível satisfazer essa checagem e depois sair dela com ../:

Bypass com pasta base no início 
filename=/var/www/images/../../../etc/passwd

# começa com a pasta esperada 
# mas sai dela em seguida

5. Extensão obrigatória no final

Se a aplicação exige que o arquivo termine em .png, um null byte %00 encerra o caminho antes da extensão ser verificada:

Bypass com null byte 
filename=../../../etc/passwd%00.png

# a aplicação vê ".png" no final 
# mas o sistema lê só até o %00
🔍 O bypass com null byte funciona em linguagens mais antigas (PHP < 5.3.4, por exemplo). Em ambientes modernos geralmente não funciona, mas vale testar.

Como prevenir

A defesa mais eficaz é simples: não passe entrada do usuário diretamente para APIs de filesystem. Sempre que possível, redesenhe a funcionalidade para que o usuário selecione de uma lista de valores fixos, sem controle sobre o caminho real do arquivo.

Quando isso não for possível, use duas camadas de defesa juntas:

  • Valide a entrada — idealmente com uma whitelist de nomes permitidos. Se não for possível, aceite somente caracteres alfanuméricos.
  • Canonicalize o caminho — monte o caminho final com a pasta base e verifique que ele ainda começa com essa pasta antes de abrir o arquivo.
Python — defesa com canonicalização 
import os
 
BASE_DIR = "/var/www/images"
 
def load_image(filename):
    # monta o caminho e resolve todos os ../
    full_path = os.path.realpath(
        os.path.join(BASE_DIR, filename)
    )
 
    # verifica se o caminho resolvido ainda está dentro da pasta base
    if not full_path.startswith(BASE_DIR + "/"):
        raise PermissionError("acesso negado")
 
    with open(full_path, "rb") as f:
        return f.read()
✅ os.path.realpath() resolve todos os ../, links simbólicos e caminhos absolutos antes da checagem. Por isso funciona contra todos os bypasses vistos acima.
