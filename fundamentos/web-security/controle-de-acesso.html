⚫ O que é controle de acesso?

Controle de acesso é o que decide quem pode fazer o quê dentro de uma aplicação. Ele trabalha junto com outros dois mecanismos, mas resolve um problema diferente de cada um:

  • Autenticação — confirma quem é o usuário (login e senha).
  • Gerenciamento de sessão — garante que as próximas requisições pertencem àquele mesmo usuário (cookie de sessão, token).
  • Controle de acesso — decide se aquele usuário, já identificado, tem permissão para fazer a ação que está tentando fazer.

Um usuário pode estar perfeitamente autenticado e ainda assim conseguir fazer coisas que não devia, como: apagar a conta de outra pessoa, ver dados de terceiros, acessar um painel de admin. Isso é controle de acesso quebrado (Broken Access Control), e é uma das vulnerabilidades mais comuns e graves que existem.

⚫ Os três tipos de controle de acesso

Vertical

Restringe funcionalidades a tipos diferentes de usuário (Ex: Administrator vs Cliente). Em um e-commerce fictício como o g1its-vulnerable-site.com, um administrador pode apagar contas de qualquer cliente; um cliente comum não pode fazer isso de forma alguma.

Horizontal

Restringe o acesso a um subconjunto de recursos do mesmo tipo. No mesmo site, por exemplo, cada cliente pode ver os próprios pedidos e dados de pagamento, mas não os pedidos de outro cliente.

Dependente de contexto

Restringe ações com base no estado atual da aplicação. Por exemplo, o carrinho de compras do g1its-vulnerable-site.com não deveria poder ser alterado depois que o pagamento já foi processado, mesmo que o próprio dono do carrinho esteja tentando mexer nele.

⚫ Escalada vertical de privilégios

Acontece quando um usuário comum ganha acesso a funcionalidades reservadas a um nível mais alto. Como um cliente comum acessando o painel de admin. Existem várias formas de explorar isso:

1. Funcionalidade sem proteção nenhuma

A forma mais simples de quebra: a aplicação simplesmente não verifica nada. O painel de admin não aparece no menu do usuário comum, mas a URL continua funcionando se você simplesmente acessá-la direto:

Sem verificação nenhuma no backend 
https://g1its-vulnerable-site.com/admin
# acesso ao endpoint de admin por meio de url conhecida

Às vezes a URL do endpoint de admin não é conhecida, mas também nem precisa ser adivinhada, ela pode aparecer sozinha em lugares como o robots.txt, ou dentro do próprio código JavaScript que monta a interface, mostrando algo como:

JavaScript que entrega a URL escondida 
const isAdmin = false;

if (isAdmin) {
  renderLink("/painel-secreto-de-administrador");
}
  # Aqui se o código não realizasse essa verificação de
  # autorização, bastaria acessar o endpoint diretamente.
🔍 Esconder uma URL é security by obscurity — segurança por obscuridade. Não é um controle de acesso, é só torná-lo mais difícil de achar.

2. Controle de acesso baseado em parâmetro

Algumas aplicações decidem o papel do usuário no login e guardam esse dado em algum lugar que o próprio usuário controla. Como um cookie, um campo hidden, ou até um parâmetro na URL:

Requisição que carrega o "papel" do usuário 
https://g1its-vulnerable-site.com/home?role=user

# o atacante pode simplesmente troca o valor:
https://g1its-vulnerable-site.com/home?role=admin

No backend, isso costuma virar um código tão simples quanto:

Python — backend confiando no parâmetro 
role = request.args.get("role")  # direto da URL, sem checar

if role == "admin":
    return render_admin_panel()

# nesse caso basta mudar a role de ?role=user para ?role=admin
# para ter acesso de administrador

3. Configuração errada na camada de plataforma

Algumas aplicações aplicam o controle de acesso no nível do servidor/proxy, restringindo método HTTP e URL. Por exemplo, uma regra de firewall pode dizer:

Regra de acesso na camada de plataforma 
DENY: POST /admin/deleteUser  →  grupo "managers"

Duas formas comuns de furar isso:

Headers que sobrescrevem a URL. Alguns frameworks aceitam headers não-padrão como X-Original-URL para reescrever a rota da requisição internamente, enquanto o firewall, que só olha a URL visível, não percebe:

Bypass via header de override de URL 
POST / HTTP/1.1
Host: g1its-vulnerable-site.com
X-Original-URL: /admin/deleteUser

# o firewall vê a requisição indo para "/" — libera
# a aplicação interpreta o header e roteia para /admin/deleteUser

Troca do método HTTP. Se a regra bloqueia só POST, tentar a mesma ação com GET pode simplesmente passar batido:

Bypass via troca de método HTTP 
GET /admin/deleteUser?id=42 HTTP/1.1

4. Discrepância na hora de comparar a URL

Front-end e back-end nem sempre concordam sobre o que é "a mesma URL". Se um lado normaliza maiúsculas/minúsculas e o outro não, dá pra escapar da regra trocando a capitalização:

Bypass via capitalização 
# bloqueado pela regra de acesso:
/admin/deleteUser

# mas isso aqui pode escapar da regra e ainda
# resolver para o mesmo endpoint internamente:
/ADMIN/DELETEUSER
/Admin/DeleteUser/

⚫ Escalada horizontal e IDOR

Escalada horizontal acontece quando um usuário acessa recursos de outro usuário do mesmo nível.

Acessando o pedido de outro cliente 
# sua própria conta:
https://g1its-vulnerable-site.com/myaccount?id=123

# trocando o id, você acessa a conta de outro cliente:
https://g1its-vulnerable-site.com/myaccount?id=124

Isso é um exemplo de IDOR — Insecure Direct Object Reference. O nome é grande, mas a ideia é simples: a aplicação usa um valor que o usuário controla (o id da URL) para buscar um recurso diretamente no banco, sem checar se aquele recurso pertence a quem está pedindo.

Python — backend vulnerável a IDOR 
user_id = request.args.get("id")
account = db.query("SELECT * FROM accounts WHERE id = ?", user_id)
return render(account)
💡 Trocar um número sequencial (123 → 124) é o caso mais óbvio. Quando a aplicação usa identificadores aleatórios (GUIDs) para dificultar isso, vale procurar onde esses GUIDs de outros usuários aparecem expostos em outro lugar do site, como em comentários, mensagens ou avaliações públicas, etc.

⚫ Outros pontos de quebra

Processos em múltiplas etapas

Funcionalidades feitas em várias etapas — preencher, revisar, confirmar — às vezes protegem só os primeiros passos e esquecem o último. Se a aplicação assume que só chega na etapa 3 quem já passou pelas etapas 1 e 2 (que são protegidas), um atacante pode pular direto para a 3 e enviar os parâmetros necessários manualmente:

Fluxo esperado

1. Carrega formulário de edição (protegido)
2. Envia alterações (protegido)
3. Confirma (sem checagem nenhuma)

Bypass do atacante

Pula direto para a etapa 3, enviando os parâmetros da etapa 2 manualmente, sem ter passado pelas etapas protegidas.

Controle baseado no header Referer

Algumas aplicações podem achar que uma sub-página de admin está protegida só porque checam se o Referer (header enviado pelo navegador que indica a URL da página web anterior de onde o usuário veio), contém a URL do painel principal:

Header Referer forjado 
POST /admin/deleteUser HTTP/1.1
Host: g1its-vulnerable-site.com
Referer: https://g1its-vulnerable-site.com/admin

# aqui a aplicação considera que pelo usuário vir de uma página    
# que só deve ser acessada pelo admin, então provavelmente o user    
# deve ser o admin, porém o atacante pode controlar o valor
# desse header. Basta colocar a URL "certa" e a checagem passa, 
# mesmo sem ter visitado o painel de admin de verdade.

Controle baseado em localização

Restrições por país ou região (comuns em bancos e serviços de streaming) costumam ser contornadas com VPN, proxy, ou simplesmente manipulando dados de geolocalização no navegador, nenhum desses métodos é controle de acesso efetivo.

⚫ Como prevenir

Não existe uma única correção mágica — controle de acesso precisa de defesa em profundidade, aplicando vários princípios juntos:

  • Nunca confie em obscuridade: Esconder uma URL não é proteção.
  • Negue por padrão: Todo recurso deveria começar bloqueado, e só ser liberado explicitamente quando fizer sentido.
  • Centralize a verificação: Use um único mecanismo de controle de acesso para a aplicação inteira, em vez de checagens espalhadas e inconsistentes.
  • Declare permissões no código, não confie no front-end: A checagem precisa acontecer no servidor, sempre, mesmo que o botão já esteja escondido na tela.
  • Audite e teste de verdade: Tente ativamente acessar recursos de outros papéis e de outros usuários antes de colocar em produção.

Um exemplo de checagem centralizada, em vez de espalhada por cada rota:

Python — checagem de acesso centralizada 
def require_owner(resource, current_user):

    # Nega por padrão. Só libera se o dono bater com o usuário logado.
    if resource.owner_id != current_user.id and not current_user.is_admin:
        raise PermissionError("acesso negado")
    return resource

# toda rota que acessa um recurso passa por aqui antes de continuar:
def view_account(account_id, current_user):
    account = db.get_account(account_id)
    account = require_owner(account, current_user)  # checagem central
    return render(account)
⚠️ A diferença chave: a checagem compara o dono real do recurso (vindo do banco) com o usuário da sessão atual, nunca com um valor que veio da própria requisição, como um id ou role na URL.
